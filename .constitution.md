# Constitución del Ecosistema de Asistencia Emocional

Esta es la constitución del sistema (Spec Kit). Todos los agentes de IA (incluyendo Diana Deep Agents) y desarrolladores humanos deben cumplir estrictamente estas reglas al modificar el repositorio `Plataforma-Emocional-`.

## 1. Ciberseguridad y Privacidad como Pilar Absoluto
1.1. **Cero Exposición PII:** Ningún agente registrará o expondrá en logs (ni del frontend ni del backend) Información de Identificación Personal (PII) o transcripciones de salud mental de los estudiantes.
1.2. **Aislamiento de la IA:** El frontend de Angular **NUNCA** se comunicará de manera directa con APIs externas comerciales de LLM. Todas las llamadas deben ir dirigidas al núcleo local (FastAPI).
1.3. **Row Level Security (RLS):** Cualquier diseño de base de datos para PostgreSQL que maneje la tabla `messages`, `chats` o `clinical_records` DEBE incluir reglas RLS. El estudiante solo puede leer lo suyo. El psicólogo solo puede leer a los asignados.
1.4. **Tokens JWT:** Todas las peticiones HTTP desde Angular hacia PostgREST o el núcleo IA deben llevar inyectado un token JWT válido (gestionado por interceptores).

## 2. La Especificación es la Ley
2.1. **Adhesión al `spec.md`:** Ningún agente debe crear nuevas vistas, funciones o tablas que no estén documentadas en `spec.md`. Si se detecta un vacío lógico, el agente debe reportarlo al humano en lugar de inventar la arquitectura ("No vibe coding").

## 3. Prioridad Clínica y Responsabilidad
3.1. **El Chat no diagnostica:** Todo componente visual interactuando con Amati debe incluir un disclaimer visible de que el bot no sustituye la terapia profesional.
3.2. **Preservación de Alertas:** El cálculo de `urgency_score` emitido por el LLM no puede ser alterado en el frontend; el frontend solo lo recibe y lo renderiza de forma visual (ej. semáforo). El psicólogo tiene la decisión final.

## 4. Diseño y UX (User Experience)
4.1. El diseño en Angular utilizará obligatoriamente los principios de **Glassmorphism**, **Dark Theme premium**, tipografías limpias y paletas de colores enfocadas a la reducción de ansiedad (tonos púrpuras, azules oscuros y acentos suaves).
4.2. Se evitarán los componentes monolíticos. Se debe usar diseño basado en componentes pequeños y escalables.

## 5. Prácticas de Desarrollo y Documentación
5.1. **Documentación Exhaustiva:** TODOS los bloques de código, funciones, servicios y componentes creados DEBEN tener comentarios explicativos detallados (docstrings en TS/JS, comentarios SQL) que describan su propósito y funcionamiento, para facilitar la lectura por parte de otros miembros del equipo.
5.2. **Preparación para Migración:** Este código se desarrolla temporalmente en paralelo en este repositorio, pero en el futuro será migrado al monorepo principal (`Ecosistema-de-Asistencia-Emocional-con-IA-Generativa`). Por lo tanto, el código debe ser altamente modular, utilizando rutas relativas limpias y evitando dependencias fuertemente acopladas al entorno actual.
